## Findings

- `v004` の結果で、今回の公開 LB では broad prompt bank より trajectory-search / domain-diverse exfil / bounded fill が明確に強いことを確認した。
- local deterministic score は `0.00` のままでも public LB `16.230` が出ており、引き続き smoke test は破損検知として扱う。
- `v005` は長時間 running だったが完走し、public `27.150` まで伸びた。public 最高スコア notebook `27.32` とほぼ同水準。
- `v006` は v005 の runtime risk を避けるため、v004 の return target `180` と search ceiling `600s` を維持し、multi-post severity multiplier を取りに行く。
- SDK 上、local evaluator の replay hop は `4`、score cell は `tool_events` 由来、`http.post` の score-cell 差分は URL domain。v006 はこの前提に合わせて endpoint counts を `(2, 3, 4)` にした。
- `sk_live_test` 単体は EXFILTRATION predicate を発火しつつ public guardrail の ultra-dangerous keyword を含まないため、fill 主力 payload にした。
- `AI Agent 25.205` notebook は v005 より低いが、`MAX_VERIFIED=120`、`MAX_TOTAL=300`、大きな static fill という短時間 aggressive-fill 構成が参考になる。
- v007 は v006 の結果待ち中の別軸 hedge。`RETURN_TARGET_CANDIDATES=300`、`MAX_VERIFIED_CANDIDATES=120`、`SEARCH_FRACTION=0.18`、`ENDPOINT_COUNTS_TO_CALIBRATE=(1, 2)`、`FILL_PAYLOADS=("sk_live_test",)`。
- Deterministic static-candidate probe では、v006/v007 とも 32/32 candidates が 4 successful `http.post` を出した。multi-post 文言はローカル deterministic では機能しているが、hosted model での成否は public score で判断する。
- v006 は `Notebook Timeout` になった。score ratio は測れないため、multipost の有効性ではなく runtime envelope の失敗として扱う。
- v008 は v006 timeout を受けて `RETURN_TARGET_CANDIDATES=96`、`MAX_VERIFIED_CANDIDATES=80`、`ENDPOINT_COUNTS_TO_CALIBRATE=(2, 3)`、`ENDPOINTS_PER_MSG=4` に下げて提出した。
- v007 も `Notebook Timeout` になった。300-candidate 構成に repeated-post 系 prompt を混ぜると v005 より runtime risk が大きい。
- v008 public `9.220` は、96 candidates が single-post exfil として成立した場合の概算 `8.64` に近い。全 96 件成立を仮定すると平均 successful post 数は約 `1.08` posts/finding で、hosted eval では multipost multiplier がほぼ乗っていない。
- v006/v007/v008 の結果から、現行 multipost prompt family は主軸から外す。次回は v005 系の single-post aggressive fill、private-safe payload、または prompt-bank 統合を優先する。
