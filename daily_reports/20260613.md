## Findings

- このコンペは通常の CSV 予測ではなく、Kaggle Notebook/script kernel が `/kaggle/working/attack.py` を生成する形式。
- Kaggle の competition submit では `-k <kernel> -v <version> -f submission.csv` を使う。
- UI submit は `attack.py` が Output に見えていても `Could not find provided output file attack.py` で失敗することがある。
- Kaggle pre-submit check のため、script kernel 実行時に placeholder `submission.csv` も出す必要がある。
- `enable_gpu: false` が安定。GPU 指定は P100 扱いになり、今回の提出では失敗要因になった。
- Local deterministic smoke score は重要な破損検知にはなるが、public LB の予測には弱い。
- `v002` の改善は、public model が直接 `http.post` exfil 系プロンプトに反応したことが主因と推定する。
- `v003` は `0.630` で、`v002` から小幅に改善した。今日の best は `v003`。
- `17-1` notebook は score `17.085` の上位実装なので、次の本命は `v004_trajectory_exfil` としてこの実装思想を移植すること。
- `v004` は提出済みだが採点待ち。狙いは `v003` の broad bank ではなく、domain-diverse exfil の bounded fill。
- `16.980` notebook は `v003` に近い broad prompt-bank 型だが、exact prompt 文言、phase 順、成功単発への aggressive mutation が強そう。`v004` が伸びなければ次はこの exact bank を優先する。
- `27.32` notebook は `17-1` の trajectory-search 実装そのもので、違いは `full` ではなく `aggressive` profile を選ぶ点。次回の最短改善は prompt bank 合成より先に `v005_trajectory_exfil_aggressive` で caps と fallback 数を上げること。
- `v005` は提出済み。v004/v005 ともに Kaggle 側の採点待ちなので、明日はまず両方の public score を確認してログを更新する。
