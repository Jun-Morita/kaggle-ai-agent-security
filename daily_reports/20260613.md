## Findings

- このコンペは通常の CSV 予測ではなく、Kaggle Notebook/script kernel が `/kaggle/working/attack.py` を生成する形式。
- Kaggle の competition submit では `-k <kernel> -v <version> -f submission.csv` を使う。
- UI submit は `attack.py` が Output に見えていても `Could not find provided output file attack.py` で失敗することがある。
- Kaggle pre-submit check のため、script kernel 実行時に placeholder `submission.csv` も出す必要がある。
- `enable_gpu: false` が安定。GPU 指定は P100 扱いになり、今回の提出では失敗要因になった。
- Local deterministic smoke score は重要な破損検知にはなるが、public LB の予測には弱い。
- `v002` の改善は、public model が直接 `http.post` exfil 系プロンプトに反応したことが主因と推定する。

## Replay / Failure Notes

- `v001` の version 1-6 は提出配線のデバッグに使った。
- separate `submission.py` 方式は Kaggle に必要ファイルが乗らず失敗した。
- `attack.py` だけを Output に出しても、competition submit 側では `submission.csv` が必要だった。
- `kaggle competitions submit -f attack.py` は `400 Bad Request` になった。
- 正しい提出コマンドは次の形:

```bash
UV_CACHE_DIR=/tmp/uv-cache \
  uv run kaggle competitions submit \
  ai-agent-security-multi-step-tool-attacks \
  -k junichiromorita/ai-agent-security-v002-public-reachable \
  -v 1 \
  -f submission.csv \
  -m "v002 public reachable"
```
