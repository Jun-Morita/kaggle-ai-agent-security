## Findings

- `V2 Exfil Mass Shift` は新しい attack class ではなく、`eda-agent-security-trajectory-search` 系の scale-up / phase shift。
- public `32.71` は `360 * (16 + 2) / 200 = 32.4` に近く、multipost ではなく single-post finding 数の増加で説明できる。
- v006/v007 timeout と v008 public `9.220` から、現行 repeated-post prompt family を続けるより、v005-style exfil fill を増やす方が期待値が高い。
- v009 の最大リスクは runtime。public notebook は完走しているが、360 candidates は v005 より重い。
- v009 は timeout したため、この環境では `360` returned candidates / `980s` search ceiling は重すぎる。v005 の `300` candidates は完走済みなので、次は `315-330` 程度の midpoint か、search ceiling を抑えた v005+small-scale で検証する。
- `AI Agent Security -(ReSubmission)` は v009 とほぼ同じ 360-candidate mass-shift 系で、新規採用点は少ない。
- `Replay-Dense Exfiltration Boundary Search` は v009 と異なり、single-endpoint / `DEFAULT_MAX_HOPS=1` / `MAX_SEARCH_S=60s` / `MAX_VERIFIED_CANDIDATES=60` にして replay-hop load を削る。public score `36` は `400 * 18 / 200 = 36` と一致し、400件前後の単発 exfil が完走した可能性が高い。
- Host discussion により、長時間 pending は必ずしも participant code の問題ではなく、attack-generation 後の replay と evaluator-side timeout enforcement の影響を受けると分かった。v011 は search を伸ばすより single-hop candidate 幅を増やす実験として位置づける。
